01 / Architecture

整体架构

对局内核 + 多控制面 + 外部决策服务。游戏后端只负责房间与对局事实； 真人、脚本、LLM Agent 与实验 runner 都通过同一套公开协议接入，规则层不区分控制器类型。

1设计取向

系统最关键的一个决策是：把「狼人杀规则」「玩家决策」「实验编排」三件事彻底分开。游戏后端是唯一的局内事实源，只维护对局发生了什么；它不理解某个玩家背后是真人、浏览器脚本、Agent Player Server 还是实验 runner。

这样拆分的收益是同一个引擎既能支撑真人体验，也能支撑批量 Agent 实验；上层入口可以各自按产品体验或实验效率演进，而不必把规则、视角隔离、Agent prompt 和实验统计揉进同一个系统。

2游戏系统只认识玩家与动作

核心对象是 room、game、player、ActionContext 与 ActionSubmit。同构接入体现在三条路径上，它们共享同一套局内动作协议。

human

真人

游戏前端 join → ready → 查询 pending action → 提交动作 → 拉取 view / events / replay。

ai backfill

AI 补位

游戏前端创建 Agent session；随后 Agent 自己 join、ready、轮询 pending action 并提交动作。

batch

批量实验

实验 runner 创建房间、创建 Agent session、等待 ready 并开局；局内动作仍由 Agent 直接提交。

因此游戏系统的职责保持稳定：它只维护「这局游戏发生了什么」，不维护「为什么某个控制器这样决策」。规则正确性与信息隔离全部在后端完成。

3两类控制面

游戏前端与实验控制台地位相近，都是游戏系统之上的控制与观察入口，差异在目标用户与工作流。二者都不拥有局内状态。

控制面面向目标主要能力不负责
游戏前端 真人游玩、手动调试、演示 登录、建/加入房间、准备、AI 补位、真人行动、观战、回放 批量统计、跨房间调度、Agent 内部推理
实验控制台 对照实验、批量迭代、诊断 多房间并行、多局串行、固定角色、公开名重排、summary、LLM trace、live/replay 狼人杀规则、局内动作代理、prompt 解析

需要房间、对局、事件、视角或 replay 时，它们回到游戏系统读取；需要 Agent 内部推理与模型调用时，实验控制台回到 Agent 数据库读取。

4身份与信息隔离

系统区分三种命名，避免自进化 Agent 从多局公开名字中反推出模型或阵营。

room_user_name

房间成员名，用于 join、ready 与按玩家轮询 pending action。可稳定绑定实验 slot，但不直接暴露为局内玩家名。

game_user_name

本局公开名，出现在游戏状态、事件、发言、投票与 action context 中。实验侧通过 public_name_pool 每局重排。

agent_identity_id

Agent 私有稳定身份，用于跨局策略记忆与实验分析，不暴露给游戏系统玩家视角。

信息隔离由游戏系统后端执行。公共视角不包含角色真相、狼人夜聊、验人结果、女巫药品、私有动作请求或 Agent raw output；玩家视角只含该玩家可见信息；上帝视角用于可信调试与实验复盘。

5数据事实源

database: werewolf_game
├── schema werewolf_app          # game-system：局内事实
└── schema werewolf_experiments  # experiments：实验编排事实

database: werewolf_agent         # agent：认知、决策、记忆与 LLM trace
数据类型归属
胜负、角色、死亡、事件、动作上下文/响应、快照、replaygame-system
实验配置、房间/对局映射、runner events、summary 引用experiments
belief、decision、postgame review、strategy memory、LLM 调用链路agent

这种归属约定避免同一局游戏在多个库中出现两份事实，也让各服务可以独立演进。replay、LLM trace 与 runner events 三者共同构成完整的复盘链路。
