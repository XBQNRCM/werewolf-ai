01 / Architecture

整体架构

对局事实源 + 多控制面 + 外部决策服务。游戏后端只维护规则、状态和可见事实； 真人、AI 玩家与实验编排器都通过同一套公开协议接入，规则层不区分控制器类型。

1设计取向

系统最关键的决策是把「狼人杀规则」「玩家决策」「实验编排」三件事拆开。游戏后端是唯一的局内事实源，负责判断当前局面、可见信息和合法动作；Agent 只基于可见状态做推理与行动；实验平台只组织多局对照和结果分析。

这样拆分后，同一套对局规则既能支撑真人体验，也能支撑批量 Agent 实验。前端可以专注交互，Agent 可以专注决策质量，实验台可以专注可复现验证，三者不会互相污染边界。

2统一动作协议

游戏系统不关心玩家背后是真人、AI 服务还是实验脚本。所有控制器都读取当前合法动作，再提交结构化行动，由后端统一校验和推进状态。

真人

真人

通过游戏前端加入房间、准备、读取待行动作、提交发言/投票/技能，并查看实时视图与回放。

AI 补位

AI 补位

前端只负责创建 AI 玩家会话；之后 Agent 像普通玩家一样加入、准备、轮询待行动作并提交决策。

批量实验

批量实验

实验编排器批量创建房间和 Agent 会话、等待就绪并开局；局内动作仍由 Agent 直接连到游戏系统提交。

这个设计让规则正确性、行动合法性和信息隔离始终由后端保证；Agent 的推理过程和实验平台的调度策略不会渗入对局规则层。

3两类控制面

游戏前端与实验控制台地位相近，都是游戏系统之上的控制与观察入口，差异在目标用户与工作流。二者都不拥有局内状态。

控制面面向目标主要能力不负责
游戏前端 真人游玩、手动调试、演示 登录、建/加入房间、准备、AI 补位、真人行动、观战、回放 批量统计、跨房间调度、Agent 内部推理
实验控制台 对照实验、批量迭代、诊断 多房间并行、多局串行、灵活实验配置、实验摘要、模型调用记录、观战与回放 狼人杀规则、局内动作代理、prompt 解析

需要房间、对局、事件、视角或回放时，控制面回到游戏系统读取；需要 Agent 内部推理与模型调用时，实验控制台回到 Agent 数据库读取。

4身份与信息隔离

系统区分三种命名，避免自进化 Agent 从多局公开名字中反推出模型或阵营。

room_user_name

房间成员名，用于 join、ready 与按玩家轮询 pending action。可稳定绑定实验 slot，但不直接暴露为局内玩家名。

game_user_name

本局公开名，出现在游戏状态、事件、发言、投票与 action context 中。实验侧通过 public_name_pool 每局重排。

agent_identity_id

Agent 私有稳定身份，用于跨局策略记忆与实验分析，不暴露给游戏系统玩家视角。

信息隔离由游戏系统后端执行。公共视角不包含角色真相、狼人夜聊、验人结果、女巫药品、私有动作请求或 Agent raw output；玩家视角只含该玩家可见信息；上帝视角用于可信调试与实验复盘。

5数据事实源

database: werewolf_game
├── schema werewolf_app          # game-system：局内事实
└── schema werewolf_experiments  # experiments：实验编排事实

database: werewolf_agent         # agent：认知、决策、记忆与模型调用记录
数据类型归属
胜负、角色、死亡、事件、动作上下文/响应、快照、回放game-system
实验配置、房间/对局映射、调度事件、实验摘要引用experiments
身份推断、行动决策、终局复盘、策略记忆、模型调用链路agent

这种归属约定避免同一局游戏在多个库中出现两份事实，也让各服务可以独立演进。对局回放、模型调用记录与实验调度事件共同构成完整的复盘链路。
