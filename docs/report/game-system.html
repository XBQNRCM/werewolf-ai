02 / Game System

游戏系统后端

推理内核：对局状态机与技能调度、服务端信息隔离、event / frame / replay， 以及「REST 确认事实、WebSocket 提示刷新」的通信机制。所有局内正确性都在后端完成。

1对局引擎与技能调度

角色技能被统一抽象为「某个阶段向某些玩家生成合法行动」。引擎不直接调用 Agent，只根据 GameState 生成 pending action，等待外部提交动作后再推进。

阶段行动者合法动作
狼人夜聊存活狼人，确定性顺序werewolf_chat，队内可见
狼人击杀存活狼人kill，允许空刀时可 skip；平票复投后按种子裁决
预言家存活预言家check，只获得阵营
女巫存活女巫且有药use_save / use_poison / skip
白天发言存活玩家按座次队列speak
白天投票存活玩家vote，允许时可 abstain
遗言 / 开枪出局玩家 / 出局猎人last_word / shoot

支持 6/8/10/12 人局与五类角色（狼人、预言家、女巫、猎人、村民），屠边胜负。房间是长期容器、可串行多局——这对 Agent 跨局复盘与实验同房多局至关重要；每局由 room_seed + game_index 派生 game_seed，可复现且不重复。

2服务端信息隔离

信息隔离不是前端隐藏字段，而是后端视角裁剪。同一套规则在事件接口、WebSocket、回放快照、动作请求与动作响应中被重复应用，避免单一路径遗漏导致泄露。

视角用途可见信息
public观战、公开回放公开玩家状态、公开事件、发言、投票与结算；夜间统一折叠为 night。
player真人/Agent 决策、玩家回放公开信息 + 自己身份与私有事件；狼人可见队友身份与狼队事件。
god调试、实验复盘完整服务端状态、全部事件、动作请求/响应与 replay frame。

四类上下文

  • 服务端上下文：GameState，仅后端或 god 暴露
  • 公共上下文：build_public_view() 裁剪生成
  • 玩家私有上下文：build_player_view() 按玩家生成
  • 决策上下文：ActionContext.visible_state

隔离不变量

  • 公共视图不出现任何私有 action request
  • 更换视图参数无法绕过身份校验
  • 回放按视角裁剪，历史数据不突破隔离
  • Agent 收到的已是裁剪结果，无需维护隐藏事实

3Event、Frame 与 Replay

持久化目标是让对局可恢复、可回放、可分析。回放不是前端缓存，而是后端根据事件、快照、动作请求与动作响应组装。

action_request  →  action_response  →  events  →  state_after

上述链路构成一个 replay frame：它完整记录了行动者当时看到什么（visible_state）、能做什么（legal_actions）、实际提交了什么（action_response），以及状态如何被推进（events + state_after）。事件本身携带 visibility 与可选 user_name，在所有输出路径统一过滤；因此同一份 frame 既适合前端播放，也适合人工排查与 Agent 复盘。

4通信机制

REST 确认事实、WebSocket 提示刷新。客户端收到推送后重新拉取完整视图，而不依赖推送消息中的局部数据自行推断状态。

Pending Action 与 Action Submit

ActionContext

  • request_id：绑定状态版本、阶段、队列位置
  • legal_actions：当前合法动作与合法目标
  • action_instruction：中文行动提示 + JSON Schema
  • visible_state：按行动者视角裁剪的上下文

提交与幂等

  • request_id + idempotency_key 共同保证幂等
  • 每次提交重新计算 pending，不信任客户端缓存
  • 过期 request_id 不命中当前 pending
  • 同一 game_id 加锁，动作串行生效

WebSocket 推送

ws://<host>/ws/rooms/{room_id}?view=public|player|god&user_name=<name>

消息类型包括 room_updated、game_started、game_updated、game_voided 等，只提示「有更新」。推送丢失时，客户端可随时通过 REST 重新获取完整状态——推送从不作为最终数据源。

5多 Agent 协调

游戏系统不内置 Agent 注册中心，也不代理 Agent 间的私下通信。协调通过状态机间接完成。

1加入房间以普通玩家身份 join
2准备自己 set_ready
3轮询GET pending-actions
4决策读取 ActionContext
5提交POST actions

狼人夜聊以 werewolf_chat 动作与 werewolf 可见性记录；白天讨论以 speak 公开发言承载；多局实验中的公开名通过 player_aliases 每局重排。无论按玩家轮询还是 scope=all 集中拉取，动作提交都经过同一个校验入口，编排不会绕过规则或信息隔离。
