02 / Game System

游戏系统后端

对局事实源：规则状态机与技能调度、服务端信息隔离、事件 / 快照 / 回放， 以及「REST 确认事实、WebSocket 提示刷新」的通信机制。所有局内正确性都在后端完成。

1房间、对局与技能调度

游戏系统先管理长期存在的房间，再在房间中串行启动一局局具体对局。每局内部，角色技能被统一抽象为「某个阶段向某些玩家生成合法行动」；引擎不直接调用 Agent，只等待外部提交动作后推进状态。

阶段行动者合法动作
狼人夜聊存活狼人，确定性顺序werewolf_chat，队内可见
狼人击杀存活狼人kill，允许空刀时可 skip；平票复投后按种子裁决
预言家存活预言家check，只获得阵营
女巫存活女巫且有药use_save / use_poison / skip
白天发言存活玩家按座次队列speak
白天投票存活玩家vote，允许时可 abstain
遗言 / 开枪出局玩家 / 出局猎人last_word / shoot

支持 6/8/10/12 人局与五类角色（狼人、预言家、女巫、猎人、村民），采用屠边胜负。房间作为长期容器，是 Agent 跨局复盘和实验同房多局的基础；每局由 room_seed + game_index 派生 game_seed，保证可复现且不会重复同一随机过程。

2服务端信息隔离

信息隔离不是前端隐藏字段，而是后端视角裁剪。同一套规则在事件接口、WebSocket、回放快照、动作请求与动作响应中被重复应用，避免单一路径遗漏导致泄露。

视角用途可见信息
public观战、公开回放公开玩家状态、公开事件、发言、投票与结算；夜间统一折叠为 night。
player真人/Agent 决策、玩家回放公开信息 + 自己身份与私有事件；狼人可见队友身份与狼队事件。
god调试、实验复盘完整服务端状态、全部事件、动作请求/响应与回放帧。

四类上下文

  • 服务端上下文：GameState，仅后端或 god 暴露
  • 公共上下文：build_public_view() 裁剪生成
  • 玩家私有上下文：build_player_view() 按玩家生成
  • 决策上下文：ActionContext.visible_state

隔离不变量

  • 公共视图不出现任何私有 action request
  • 更换视图参数无法绕过身份校验
  • 回放按视角裁剪，历史数据不突破隔离
  • Agent 收到的已是裁剪结果，无需维护隐藏事实

3事件、快照与可回放链路

持久化目标是让对局可恢复、可回放、可分析。游戏系统不仅保存最终状态，也保存“谁在什么视角下做了什么，以及这次动作如何改变局面”。

Event：发生过的事实

事件是追加式事实流，记录阶段切换、发言、投票、技能结果、死亡和胜负等变化。每条事件带可见性，因此同一局可以裁剪成公开视角、玩家视角或复盘视角。

Snapshot：某一刻的完整状态

快照保存动作结算后的服务端状态，包括阶段、存活、角色、技能使用、投票容器等。它让系统可以恢复对局，也能核对事件流是否正确推进了状态。

Action Request / Response：决策边界

动作请求记录行动者当时看到的可见状态和合法动作；动作响应记录实际提交的行动、是否被接受以及产生的事件范围。它回答“当时能做什么”和“最后做了什么”。

Replay Frame：可复盘片段

回放帧把一次动作请求、动作响应、产生事件和结算后状态串在一起。前端用它播放对局，实验和 Agent 复盘用它追踪决策输入与后果。

action_request  →  action_response  →  events  →  state_after

这条链路保证回放不是前端缓存，而是从服务端事实重建出来的决策过程。因为事件、动作请求和回放帧都会按视角裁剪，同一份底层数据既能安全给观众看，也能在可信复盘中还原完整上下文。

4通信机制

REST 确认事实、WebSocket 提示刷新。客户端收到推送后重新拉取完整视图，而不依赖推送消息中的局部数据自行推断状态。

待行动作与动作提交

ActionContext

  • request_id：绑定状态版本、阶段、队列位置
  • legal_actions：当前合法动作与合法目标
  • action_instruction：中文行动提示 + JSON Schema
  • visible_state：按行动者视角裁剪的上下文

提交与幂等

  • request_id + idempotency_key 共同保证幂等
  • 每次提交重新计算 pending，不信任客户端缓存
  • 过期 request_id 不命中当前 pending
  • 同一 game_id 加锁，动作串行生效

WebSocket 推送

ws://<host>/ws/rooms/{room_id}?view=public|player|god&user_name=<name>

消息类型包括 room_updated、game_started、game_updated、game_voided 等，只提示「有更新」。推送丢失时，客户端可随时通过 REST 重新获取完整状态——推送从不作为最终数据源。

5多 Agent 协调

这里的“协调”不是中心控制器替 Agent 做决定，而是游戏状态机按规则分发行动机会；每个 Agent 只根据自己的可见上下文行动。

1加入房间以普通玩家身份 join
2准备自己 set_ready
3轮询GET pending-actions
4决策读取 ActionContext
5提交POST actions

一局游戏推进时，后端只暴露当前玩家真正可执行的动作：轮到狼人夜聊，就只让狼人提交队内可见的 werewolf_chat；轮到白天发言，就让存活玩家提交公开的 speak；轮到投票或技能，也只给出当前合法目标。Agent 之间的协作因此发生在游戏规则允许的发言、夜聊、投票和技能动作里，而不是通过额外的私有通道。

实验平台可以集中查看哪些玩家正在等待行动，但它不能替游戏系统判定合法性。无论是单个 Agent 自己轮询，还是实验侧批量调度，最终动作都必须提交到同一个后端入口，由同一套规则校验、信息隔离和状态推进逻辑处理。
