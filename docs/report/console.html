实验台设计

批量编排与观测平台：稳健的多房间实验调度、断点续跑与自愈、实时监控与错误诊断、 细粒度回放（含 LLM trace）以及多切面离线分析脚本。它只编排，不复制局内事实。

1定位与边界

实验台的边界很窄：不代理局内 action、不中转 pending action、不复制完整局内事实。胜负与 replay 以 game-system 为准；认知与 LLM trace 以 agent DB 为准；实验库只保存调度记录与跨系统 ID。

负责

编排与观测

  • 实验配置与 spec 快照
  • 多房间并行、多局串行调度
  • Agent session 通知与 ready 等待
  • 固定角色 + 公开名重排
  • runner events、summary、错误抽屉
  • 观战/回放入口与 LLM trace 查询
不负责

交还给其他服务

  • 狼人杀规则与状态机（game-system）
  • 局内 pending action 代理
  • Agent prompt 解析（agent）
  • 复制局内事件 / 快照 / raw output
  • 强制终止进行中的游戏

2Runner 编排流程

1创建实验POST experiments · 存 spec
2创建房间game-system POST rooms
3通知 AgentPOST sessions
4等待 ready轮询成员状态
5开局aliases + role_overrides
6等待结束轮询 winner
7下一局复盘后再 ready

房间是实验调度单元：一个实验房间映射一个游戏系统房间，可连续运行多局。max_parallel_rooms 控制并发；Agent 直接轮询/提交动作，runner 只负责开局与记录完成状态。固定角色通过 fixed_role 展开为每局的 alias-keyed role_overrides，配合 public_name_pool 实现「角色固定、公开名每局重排」。

3断点续跑与自愈

长时间批量实验不可避免遇到网络波动、Agent 超时或进程重启。runner 内置多条恢复路径，使实验可以从中断点继续，而不是从头重跑。

房间恢复

已创建游戏房间但实验库未写入 room id 时，通过 room_info.experiment_room_id 从游戏系统房间列表找回，不重复建房。

对局恢复

实验记录 running 但游戏已结束则补齐完成状态；检测到僵尸进行中游戏则 void_game 后重开。

Agent 恢复

旧 session 已停止或失效时，先停止残留 session，再重新通知 Agent 创建新 session 继续剩余局数。

失败、恢复、作废与清理都写入 runner_events，供前端错误抽屉与排查使用。取消实验仅停止后续调度；已进入 in_game 的对局仍由游戏系统与 Agent 自然完成，不被强杀。

4实时监控与诊断

实验详情页

  • 启动 / 暂停 / 恢复 / 取消
  • 房间矩阵：状态、完成局数、Agent slot 状态
  • 对局列表：进入观战、回放或错误抽屉
  • Runner 事件：按全部 / 错误 / 房间 / 对局过滤
  • 导出实验配置与 summary JSON

错误抽屉

  • 聚合目标对象、后端 ID 与最近错误
  • 关联 runner events 上下文
  • 实验级 error_summary 与房间级 last_error
  • Agent 通知 / 加入状态追踪

5细粒度回放与 LLM Trace

GameViewer 直接访问游戏系统接口，不经实验后端代理局内数据。god 视角可联动 Agent 数据库中的 llm_calls，实现 case-by-case 的决策分析。

模式能力
Live游戏系统 /view、/events、/pending-actions + WebSocket；pending inspector 展示 ActionContext 详情
Replay按 replay frame 查看 action context、visible state、action response、events、state_after
过滤按 phase、actor、action type、event type、raw output、invalid action 过滤；可一键跳到下一条非法动作
LLM Tracegod 视角 + Agent DB：查看每次 belief/decision/review 的真实 request/response payload

完整复盘链路：game-system replay frame（局内事实）+ agent llm_calls（模型输入输出）+ runner_events（调度日志），三者对齐即可逐帧还原「模型看到什么、做了什么、造成什么结果」。

6分析脚本

werewolf-experiments/analysis 以实验库为入口，回到 game-system replay 与 agent DB，从多个切面导出指标与图表。实验报告中的图表均由这些脚本生成。

产物内容
game_summary.csv每局胜负、天数、frame 数
vote_metrics.csv投票一致性、投中狼人率、狼人暴露率
belief_metrics.csv身份推断准确率（含 masked 口径）
memory_metrics.csvmemory 写入、helpful 率、score 变化
decision_memory_metrics.csv每次决策的 memory 选用与 fallback 情况
postgame_review_metrics.csv复盘成功率、belief/decision 错误计数

分析维度覆盖：按 game_index 的学习曲线、按 day/phase 的推断异质性、按角色的 memory 创建与复用、按 room 的独立重复轨迹。多口径并用（按局 / 按房间 / 按决策）使「闭环是否真的发生」可被分别验证。
